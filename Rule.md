
## Mini-Bili v1.0 工程规则（Rule）

**版本**：v1.0
**最后更新**：2026-05-12
**依赖文档**：Mini-Bili v1.0 SPEC

### 关于 Rule 的说明

本文档是项目的"研发制度"，告诉 AI 什么事绝对不能乱来。Rule 只是**软约束**，不是硬门禁——后续需要 Scripts（脚本）来强制执行。

随着项目演进，Rule 可能会增多。当某条 Rule 反复被 AI 违反时，应考虑将其下沉为 Scripts，变成可执行的硬检查。

### 一、数据库与数据安全

| 编号       | 规则                               | 说明                                                         |
| :--------- | :--------------------------------- | :----------------------------------------------------------- |
| **R-DB-1** | **严禁硬编码敏感信息**             | 数据库连接字符串、OSS 密钥（Bucket `mini-bili`）、JWT Secret、RabbitMQ 连接信息、敏感词库路径等敏感配置，必须通过环境变量读取，环境变量缺失时从配置文件读取。严禁直接写在代码中。 |
| **R-DB-2** | **严禁拼接 SQL**                   | 所有数据库查询必须使用 GORM 的参数化查询或 `db.Where()` 等安全方法，严禁使用 `db.Raw()` 拼接用户输入。 |
| **R-DB-3** | **数据库结构变更必须通过迁移脚本** | 所有 `CREATE TABLE`、`ALTER TABLE` 等操作必须写在迁移脚本中（如 GORM AutoMigrate 或独立 SQL 文件），严禁在业务代码中直接建表或手动修改线上数据库。 |
| **R-DB-4** | **核心查询字段必须建立索引**       | 以下字段在对应的数据库表中**必须**显式声明索引：- `play_count`（视频播放量，用于排序）- `created_at`（创建时间，用于排序）- `user_id`（用户ID，用于关联查询）- `video_id`（视频ID，用于关联查询） |

### 二、API 设计与兼容

| 编号        | 规则                                        | 说明                                                         |
| :---------- | :------------------------------------------ | :----------------------------------------------------------- |
| **R-API-1** | **必须遵循统一响应格式**                    | 所有对外 HTTP 接口必须使用以下 JSON 格式返回。**严禁**使用任何其他结构的响应体。`code` 使用业务错误码（整型），`msg` 通过 S-006 定义的错误码映射表获取，`data` 在无数据时返回 `null`。具体返回样例：`c.JSON(http.StatusOK, gin.H{ "code": code, "msg": e.GetMsg(code), "data": data, })` |
| **R-API-2** | **必须遵循 RESTful 规范**                   | 所有对外 HTTP 接口 URL 和 HTTP 方法必须遵循 RESTful 风格（NF-8）：- GET 用于查询，POST 用于创建，PUT/PATCH 用于更新，DELETE 用于删除。- URL 使用复数名词（如 `/videos`），不使用动词。 |
| **R-API-3** | **严禁破坏已有接口**                        | 已发布的 API 接口（v1.0）严禁修改已有字段名、类型或删除字段（BC-1）。只能新增字段或新增接口。 |
| **R-API-4** | **所有写操作接口及 WebSocket 连接必须鉴权** | 任何涉及数据增删改的接口（上传、发弹幕、评论、点赞、删除等），必须在 Gin 中间件层强制校验 JWT Access Token（F1）。WebSocket 连接建立时同样必须校验 JWT Access Token，校验流程必须遵循 S-011。校验失败必须返回 401，不得继续执行后续逻辑。 |

### 三、认证与安全

| 编号         | 规则                        | 说明                                                         |
| :----------- | :-------------------------- | :----------------------------------------------------------- |
| **R-AUTH-1** | **Token 安全策略**          | JWT 鉴权必须采用双 Token 机制，颁发、刷新、失效流程必须严格遵循 S-009。 |
| **R-AUTH-2** | **密码存储必须使用 bcrypt** | 任何用户密码相关逻辑，**必须且只能**使用 `golang.org/x/crypto/bcrypt` 进行哈希和校验（F1）。严禁使用 MD5、SHA256 等任何其他方式存储密码。 |

### 四、核心业务逻辑

| 编号        | 规则                                   | 说明                                                         |
| :---------- | :------------------------------------- | :----------------------------------------------------------- |
| **R-BIZ-1** | **弹幕冷却必须双端校验**               | 弹幕发送的 5 秒冷却校验逻辑必须严格遵循 S-007。严禁只依赖前端拦截。 |
| **R-BIZ-2** | **评论删除必须级联删除**               | 删除一条评论时，必须同步删除其所有层级的子评论（F7），删除流程必须遵循 S-008。严禁出现父评论已删除、子评论残留的"孤儿"数据。 |
| **R-BIZ-3** | **封面图必须校验格式和大小**           | 上传封面时，校验流程必须遵循 S-005。不符合要求必须返回明确的错误提示，拒绝整个请求。 |
| **R-BIZ-4** | **视频列表排序必须严格遵守多级优先级** | 首页视频列表必须严格按照"播放量降序 → 上传时间降序 → 弹幕数降序"的优先级返回数据（F10）。**严禁不按上述优先级排序**。 |
| **R-BIZ-5** | **弹幕敏感词过滤必须使用可配置词库**   | 弹幕内容审核（F5）**必须**从外部词库文件加载敏感词列表，词库文件路径必须通过环境变量或配置文件读取（R-DB-1）。**严禁**将敏感词硬编码在代码中。词库为空或文件不存在时，**必须**拒绝所有弹幕发送（宁可误拦，不可漏放），并记录 Error 级别日志。 |
| **R-BIZ-6** | **评论删除权限必须校验**               | UP主可删除自己视频下的**任意评论**。普通用户**只能**删除自己发布的评论。后端在接收到删除请求时**必须先校验权限**，严禁仅依赖前端按钮隐藏。权限校验失败必须返回 403。 |
| **R-BIZ-7** | **弹幕颜色必须校验十六进制格式**       | 弹幕发送时，后端必须校验 `color` 字段的格式。必须符合 `#[0-9A-Fa-f]{6}` 正则表达式。格式非法时返回错误码 `40014`，拒绝发送。不得默认填充或静默修正。 |
| **R-BIZ-8** | **用户头像必须校验格式与大小**       | 上传头像（F0）时，扩展名与 MIME 侧校验流程须与 Skill **S-005**（封面）一致：仅允许 **JPEG、PNG、GIF、BMP、WEBP**；单文件大小 **≤ 5 MB**（封面为 10 MB，头像单独上限）。不符合要求须返回明确业务错误码（`40015` / `40016`），并拒绝请求。 |

### 五、开发流程与质量底线

| 编号        | 规则                               | 说明                                                         |
| :---------- | :--------------------------------- | :----------------------------------------------------------- |
| **R-DEV-1** | **改完代码必须可运行**             | 每次代码修改完成后，必须执行 S-001（编译验证）。严禁提交无法编译或运行时报 `panic` 的代码。 |
| **R-DEV-2** | **处理完文件必须清理**             | 上传和转码过程中产生的所有本地临时文件（原始视频、转码中间产物、临时封面），在任务结束后必须删除（F2）。严禁让临时文件长期占用服务器磁盘。 |
| **R-DEV-3** | **错误必须提供明确信息**           | 任何接口返回错误时，`msg` 字段必须包含人类可读的、能帮助调用方定位问题的描述。错误码必须通过 S-006 定义的错误码映射表获取对应消息。严禁只返回"error"或空字符串。 |
| **R-DEV-4** | **转码失败重试次数限制**           | 转码任务重试逻辑必须严格遵循 S-004。严禁无限重试。           |
| **R-DEV-5** | **Redis 连接必须配置超时和连接池** | Redis 客户端初始化时必须显式设置：- 连接超时时间（DialTimeout）- 读/写超时时间（ReadTimeout / WriteTimeout）- 连接池最大连接数（PoolSize）严禁使用零值默认配置。 |
| **R-DEV-6** | **消息队列必须使用 RabbitMQ**      | 视频转码异步任务的消息队列技术选型确定为 **RabbitMQ**。所有异步任务的投递和消费必须基于 RabbitMQ 实现。严禁使用 Redis List 或其他消息队列替代。RabbitMQ 连接信息必须通过环境变量读取（R-DB-1）。 |

### 六、可观测性

| 编号        | 规则                              | 说明                                                         |
| :---------- | :-------------------------------- | :----------------------------------------------------------- |
| **R-OBS-1** | **严禁使用 fmt.Println 打印日志** | 日志初始化必须严格遵循 S-003。任何业务日志**必须**使用 zap 结构化日志库输出。`fmt.Println` 仅允许在本地调试时临时使用，**严禁进入版本控制**。 |
| **R-OBS-2** | **日志必须区分级别**              | `Info` 级别用于记录关键业务流程（如"用户登录成功"、"视频转码完成"）。`Error` 级别用于记录异常和错误（如"数据库连接失败"、"OSS 上传超时"、"敏感词库加载失败"）。严禁所有日志使用同一级别。 |

### 七、兼容与架构约束

| 编号         | 规则                                 | 说明                                                         |
| :----------- | :----------------------------------- | :----------------------------------------------------------- |
| **R-ARCH-1** | **代码模块化，预留微服务拆分能力**   | 虽然 v1.0 使用 Gin 单体架构（NF-6），但代码必须按功能拆分目录（如 `internal/handler/`、`internal/service/`、`internal/dao/`），服务间调用通过接口而非直接引用，为后续拆分到 Kratos 微服务预留空间（BC-2）。 |
| **R-ARCH-2** | **视频状态字段必须预留且仅限定义值** | 视频 `status` 字段必须使用以下**五种状态值构成的闭集**，严禁新增未定义状态：`processing`（处理中）、`published`（已发布）、`failed`（转码失败）、`pending_review`（待审核，预留）、`rejected`（审核驳回，预留）。数据库字段长度必须支持所有五种值（F2-b、BC-2）。 |

### 八、前端与交互约束

#### 1. 样式与结构保护

| 编号       | 规则                   | 说明                                                         |
| :--------- | :--------------------- | :----------------------------------------------------------- |
| **R-FE-1** | **CSS 样式文件冻结**   | `src/assets/` 和 `src/components/` 中所有包含 Tailwind CSS 类名的文件视为**只读**。AI 严禁修改已有样式，但可以在**新增组件**中使用项目已有的 Tailwind 原子类组合。 |
| **R-FE-2** | **页面骨架与路由冻结** | 路由配置（Vue Router）和页面整体 HTML 层级结构（如 Flex/Grid 布局容器）已由人类定义，AI 严禁删除或重构。AI 允许在页面中**预留的容器区域**（如 `<div id="comment-section">`）内新增功能组件。 |

#### 2. 功能增强与逻辑开发（开放区）

| 编号       | 规则                     | 说明                                                         |
| :--------- | :----------------------- | :----------------------------------------------------------- |
| **R-FE-3** | **允许完善演示按钮功能** | 前端现有的演示按钮（如"发送弹幕"、"提交评论"、"点赞"等）**允许**绑定真实的 API 调用和交互逻辑。AI 必须保持按钮的**现有样式不变**，仅补充 `@click` 事件、数据绑定和状态管理逻辑。 |
| **R-FE-4** | **允许新增功能组件**     | AI 可以在视频播放页、消息中心等页面中**新增**功能组件（如评论区列表、通知列表、弹幕发送框），但必须：- 使用项目已有的 Tailwind CSS 类名。- 放在页面中已预留的容器区域内。- 不修改已有页面骨架结构。 |

#### 3. 数据交互与逻辑隔离

| 编号       | 规则                                      | 说明                                                         |
| :--------- | :---------------------------------------- | :----------------------------------------------------------- |
| **R-FE-5** | **API 调用必须使用 TypeScript Interface** | 前端调用后端 API 时，必须使用 `axios` 或 `fetch` 封装请求。必须根据后端返回的 JSON 结构生成 TypeScript Interface（Type Guards），严禁使用 `any` 类型。 |
| **R-FE-6** | **必须处理 Loading 和 Error 状态**        | 任何异步请求（上传、登录、获取列表），必须在 UI 上反馈 `loading` 状态（如按钮变色、加载动画）和 `error` 提示（如 Toast 弹窗）。严禁只调用接口不处理用户反馈。 |
| **R-FE-7** | **必须使用环境变量配置请求地址**          | 前端请求的后端 BaseURL 必须通过环境变量（如 `.env.production`）注入，严禁硬编码 `localhost:8080` 或具体的 IP 地址。 |

### 九、文档同步

| 编号       | 规则                           | 说明                                                         |
| :--------- | :----------------------------- | :----------------------------------------------------------- |
| **R-DOC-1** | **中英 README 必须同步更新**   | 修改 `README.md`（中文）时，必须同步更新 `README_EN.md`（英文），确保描述内容、技术栈徽标、链接等保持一致。反之亦然。 |
| **R-DOC-2** | **代码变更必须同步更新文档**   | 每次修改代码（新增/删除功能、调整接口、变更依赖、修改配置项等），必须检查并同步更新相关 Markdown 文档（`README.md`、`SPEC.md`、`docs/` 目录下的所有 `.md` 文件、`deploy/DEPLOY.md` 等），确保文档描述与代码实现一致。禁止代码已改但文档还停留在旧版本的"文档腐烂"。 |
| **R-DOC-3** | **ARCHITECTURE 中英文必须完全同步** | 修改 `docs/ARCHITECTURE.md`（英文）或 `docs/ARCHITECTURE_EN.md`（中文）任一文件时，必须同步更新另一文件，确保章节结构、图表（Mermaid）、表格、代码块完全一致，仅语言不同。禁止出现一方有某章节/某图表而另一方缺失的情况。 |
| **R-DOC-4** | **Git 提交信息必须使用英文**   | `git commit -m` 的提交描述必须使用纯英文，遵循 conventional commits 格式（如 `feat:`、`fix:`、`docs:`、`refactor:`、`chore:` 等）。禁止在 commit message 中出现中文。 |
