## Mini-Bili v1.0 设计规格文档（SPEC）

**版本**：v1.0
**最后更新**：2026-05-12
**性质**：确定性需求规格

---

### 一、版本目标与范围

#### 1.1 核心目标
本版本必须交付以下三个核心能力：
1.  **用户认证与视频管理**：用户能注册、登录，**维护个人信息（用户名、头像、密码）与「我的视频」列表**，上传任意常见格式视频，系统在服务器本地异步转码为 H.264 MP4 后上传至 OSS，管理视频状态和列表，支持自定义封面上传。
2.  **实时弹幕系统**：用户在观看视频时，能发送弹幕，并能实时（延迟 ≤ 200ms）看到同视频其他用户发送的弹幕。
3.  **多级评论系统**：用户在视频下方能发表和回复评论，支持最多3级嵌套。点赞操作触发通知，通知以聚合形式展示在独立的消息中心页面。

#### 1.2 明确排除（非 v1.0 目标）
以下功能不在本版本开发范围内：
- 不做视频推荐算法。
- 不做直播功能。
- 不做用户私信、动态、关注等社交功能。
- 不做视频搜索功能，视频仅能通过列表浏览或输入视频ID访问。
- 不做支付、会员、充电等任何商业化功能。
- 不使用CDN分发，所有服务部署在单台服务器上。
- 不做后台管理系统。
- 不做视频内容审核功能。

### 二、功能需求

#### F0：用户个人信息管理（认证延伸，支撑「用户认证与视频管理」）

| 需求项         | 规格 |
| :------------- | :--- |
| 获取当前用户信息 | `GET /api/v1/users/me`，需携带有效 Access Token。响应 `data`：`user_id`（uint）、`username`、`avatar_url`（字符串，未设置时为空字符串）、`created_at`（`YYYY-MM-DD HH:MM:SS`）。 |
| 修改用户名     | `PUT /api/v1/users/me`，Body JSON：`{ "username": "..." }`。用户名规则与注册一致（3–32 个字符，支持中文、字母、数字、下划线）。若与**其他用户**重名，返回业务码 **`40006`**（`CodeUsernameExists`）。成功返回 `user_id`、`username`。 |
| 修改头像       | `POST /api/v1/users/me/avatar`，`multipart/form-data`，字段名 **`avatar`**。格式仅限 **JPEG、PNG、GIF、BMP、WEBP**；单文件 **≤ 5 MB**（Rule **R-BIZ-8**，校验流程同 Skill **S-005** 封面扩展名与大小逻辑，大小上限为 5MB）。上传至 OSS Bucket `mini-bili`，对象键 **`avatars/{user_id}.{ext}`**（见 NF-7）。成功响应 `data.avatar_url`。OSS 未配置时返回 `50000`。 |
| 修改密码       | `PUT /api/v1/users/me/password`，Body JSON：`{ "old_password", "new_password" }`。新密码长度须 **≥ 8**；存储须 **bcrypt**（R-AUTH-2）。旧密码校验失败返回 HTTP **403**，业务码 **`40301`**（`CodePasswordMismatch`，与通用无权限 `40300` 区分）。 |
| 我的视频列表   | `GET /api/v1/users/me/videos`（F2-b）：返回本人所有状态稿件，含 `failed` 与 **`fail_reason`**，支撑 AC-3 个人中心失败展示。 |

#### F1：用户认证

| 需求项   | 规格                                                         |
| :------- | :----------------------------------------------------------- |
| 注册     | 用户必须提供用户名和密码。系统对用户名做唯一性校验，重复则返回错误。 |
| 登录     | 使用用户名和密码登录。成功后服务端返回 JWT Token，作为后续鉴权接口的凭证。 |
| 密码存储 | 密码必须使用 bcrypt 算法哈希后存储，严禁明文存储。           |
| 鉴权     | 所有涉及用户数据修改的接口（**获取/修改本人信息、头像、密码**、上传视频、上传封面、发弹幕、发评论、点赞、删除评论），必须在请求头 `Authorization: Bearer <token>` 中携带有效 JWT Token，由中间件层强制校验。校验失败接口返回 401。 |

---

#### F2：视频上传

| 需求项         | 规格                                                         |
| :------------- | :----------------------------------------------------------- |
| 前置条件       | 用户必须登录。                                               |
| 请求格式       | 前端使用 `multipart/form-data` 格式上传。必须包含：<br>- `file`: 视频文件<br>- `title`: 视频标题（1-80 个字符）<br>- `description`: 视频简介（0-2000 个字符）<br>可选包含：<br>- `cover`: 自定义封面图文件。格式仅限 **JPEG、PNG、GIF、BMP、WEBP**；大小 **≤ 10 MB**。 |
| 文件格式       | 不限制用户上传的视频格式。所有格式均接受。                   |
| 文件大小       | 单视频文件必须 ≤ 500 MB，否则返回 400 错误。                 |
| 视频时长       | 单视频时长必须 ≤ 30 分钟，否则返回 400 错误。                |
| 封面上传校验   | 若用户提交了 `cover` 文件但格式或大小不符合要求，接口必须返回明确的错误信息（如"封面格式不支持"或"封面大小超过 10MB"），**拒绝整个上传请求**，要求用户修正后重新提交。 |
| 上传后处理     | 上传成功后，原始视频文件暂存至服务器本地临时目录。若用户提交了合规的 `cover` 文件，同步暂存至本地临时目录。系统立即创建一条视频记录，状态字段设置为 `processing`，同时记录视频时长和上传者信息。然后将视频 ID、本地原始文件路径及封面文件路径（如有）投递至异步转码队列。上传接口在投递完成后立即返回 201，不等待转码结果。 |
| 转码与上传逻辑 | 异步消费者从队列中取出任务，按以下顺序执行：<br>1. 从本地路径读取原始视频，使用 FFmpeg 转码为 H.264 编码的 MP4 文件，输出至本地临时目录。<br>2. 封面处理：<br>   - 若用户提交了自定义封面，校验格式和大小后直接使用。<br>   - 若用户未提交封面，使用 FFmpeg 截取视频第 1 秒帧，输出为 JPEG 图片作为默认封面。<br>3. 将转码后的 MP4 文件上传至阿里云 OSS 的 `mini-bili` Bucket，路径为 `videos/{video_id}.mp4`，记录 OSS URL。<br>4. 将封面图上传至阿里云 OSS 的 `mini-bili` Bucket，路径为 `covers/{video_id}.{ext}`（ext 为实际格式后缀），记录 OSS URL。<br>5. 更新视频记录：状态改为 `published`，写入转码视频 URL 和封面 URL。<br>6. 无论成功或失败，执行结束后必须删除本地临时目录中本次任务的所有文件。<br>转码失败时，视频状态改为 `failed`，记录失败原因，仅执行步骤6清理。 |

---

#### F2-b：视频状态管理

| 需求项     | 规格                                                         |
| :--------- | :----------------------------------------------------------- |
| 状态定义   | 一条视频记录的状态字段，定义以下五种状态值。其中 `pending_review` 与 `rejected` 为预留字段，v1.0 版本不实际使用。<br>- `processing`：处理中<br>- `published`：已发布<br>- `failed`：转码失败<br>- `pending_review`：待审核（预留，v1.0 不使用）<br>- `rejected`：审核驳回（预留，v1.0 不使用） |
| 可见性规则 | 视频列表和视频详情页，对外部普通用户仅能展示和访问状态为 `published` 的视频。<br>上传者本人，可以在其个人中心查看自己所有状态的视频，并能看到失败原因字段。 |
| 失败处理   | 转码失败时，必须在视频记录中存储失败原因。上传者查看时，前端展示"视频处理失败：{失败原因}"。 |

#### F3：视频信息展示与封面

| 需求项     | 规格                                                         |
| :--------- | :----------------------------------------------------------- |
| 展示内容   | 在视频播放页面，展示以下信息：<br>- 视频标题<br>- 视频简介<br>- 播放量<br>- 弹幕数量<br>- 评论数量<br>- 视频时长<br>- UP主<br>- 视频创建时间（格式：`YYYY-MM-DD HH:MM:SS`） |
| 播放量策略 | 播放量计数器增量存储在 Redis 中（使用 `INCR` 命令），每 10 秒刷新一次到 MySQL。页面展示的播放量从 Redis 读取。 |
| 默认封面   | 视频转码过程中，若用户未提交自定义封面，系统自动截取视频第 1 秒帧作为默认封面，上传至 OSS 的 `mini-bili` Bucket，路径为 `covers/{video_id}.jpg`。 |
| 自定义封面 | 用户可在上传视频时一并提交自定义封面图（通过 F2 的 `cover` 字段）。视频发布后，也可通过封面修改接口上传新的封面图替换。封面图要求：<br>- 格式仅限 JPEG、PNG、GIF、BMP、WEBP<br>- 大小 ≤ 10 MB<br>- 上传后上传至 OSS 的 `mini-bili` Bucket，路径为 `covers/{video_id}.{ext}`，替换原有封面 URL |

#### F4：视频播放

| 需求项   | 规格                                                         |
| :------- | :----------------------------------------------------------- |
| 播放源   | 前端播放器来源为存储在阿里云 OSS `mini-bili` Bucket 中 `videos/` 目录下的转码后 H.264 MP4 文件的 OSS 访问 URL。 |
| 播放器   | 前端使用 HTML5 `<video>` 原生播放器，不引入额外解码库。      |
| 控制功能 | 必须支持：播放/暂停、拖动进度条、音量调节、静音、全屏。      |

#### F5：弹幕发送

| 需求项   | 规格                                                         |
| :------- | :----------------------------------------------------------- |
| 前置条件 | 用户必须登录。                                               |
| 发送格式 | 用户可以在视频播放的任意时刻发送弹幕。请求必须指定：<br>- `content`: 弹幕文本（1-100 个字符）<br>- `color`: 颜色，支持十六进制色号（如 `#FF6B6B`），格式必须为 `#` 后跟 6 位十六进制字符（`0-9A-Fa-f`），不区分大小写。可选预设值：`#FFFFFF`、`#FF0000`、`#00FF00`、`#0000FF`、`#FFFF00`。<br>- `type`: 位置（仅限 `scroll` 滚动、`top` 顶部、`bottom` 底部）<br>- `video_time`: 弹幕在视频时间轴上的位置（浮点数，单位秒） |
| 发送冷却 | 同一用户在同一视频内，发送一条弹幕成功后，**前端弹幕发送按钮必须立即变灰（不可点击），并显示 5 秒倒计时**。倒计时归零后按钮恢复可点击状态。**后端同样校验冷却时间**，若冷却期内尝试发送，接口返回错误提示"发送过于频繁，请稍后再试"。 |
| 内容审核 | 弹幕内容必须经过敏感词过滤。命中敏感词库中任一词汇，后端返回错误并拒绝入库和广播。 |
| 存储     | 弹幕成功发送后必须持久化到 MySQL 数据库。                    |

---

#### F6：弹幕实时显示

| 需求项   | 规格                                                         |
| :------- | :----------------------------------------------------------- |
| 通信协议 | 前端与后端弹幕服务之间必须使用 WebSocket 长连接。严禁使用 HTTP 短轮询。 |
| 延迟指标 | 弹幕从发送方点击发送，到接收方前端屏幕渲染，总延迟在本地开发环境必须 ≤ 200ms。 |
| 历史弹幕 | 用户打开一个视频时，后端通过该 WebSocket 连接，首先一次性推送该视频最近 200 条历史弹幕至前端。前端按弹幕的 `video_time` 在对应时间点渲染。 |
| 渲染引擎 | 前端必须使用 Canvas 或 WebGL 渲染弹幕。多弹幕在同一轨道上时必须有间距，不可重叠。 |

#### F7：评论发布与删除

| 需求项   | 规格                                                         |
| :------- | :----------------------------------------------------------- |
| 前置条件 | 用户必须登录。                                               |
| 一级评论 | 用户可在视频下方直接发布评论，内容包括纯文本（1-1000 个字符）。 |
| 修改     | 禁止。任何评论一旦发布，不允许编辑。                         |
| 删除权限 | **UP主**可以删除自己视频下的任意评论。<br>**普通用户**只能删除自己发布的评论。 |
| 删除级联 | 一条评论被删除时，其下的所有子评论（所有层级）**同步删除**。被删除评论的所有数据从数据库中移除，前端不再展示。 |

---

#### F8：评论回复

| 需求项       | 规格                                                         |
| :----------- | :----------------------------------------------------------- |
| 嵌套层级     | 评论最多支持 3 级嵌套。<br>- 直接回复视频：1 级<br>- 回复 1 级评论：2 级<br>- 回复 2 级评论：3 级 |
| 超出层级处理 | 当用户回复一条 3 级评论时，新创建的评论同样作为 3 级评论处理，其父评论指向被回复的那条 3 级评论。前端不做禁止发送限制。 |
| 修改与删除   | 同 F7 规则：禁止修改；UP主可删除任意回复，普通用户仅可删除自己的回复；删除时级联删除所有子回复。 |

---

#### F9：评论点赞与通知

| 需求项   | 规格                                                         |
| :------- | :----------------------------------------------------------- |
| 交互逻辑 | 同一用户对同一条评论只能点赞一次，再次点击取消点赞。         |
| 通知触发 | 用户 A 对用户 B 的评论点赞时，系统检查该评论是否已有未读的点赞聚合通知：<br>- 如果存在一条未读的点赞聚合通知（同一评论ID），则将用户 A 追加到该通知的点赞用户列表中，更新点赞总数，不创建新通知。<br>- 如果不存在，则创建一条新的聚合通知。 |
| 通知格式 | 单条点赞聚合通知的展示格式为：<br>- 1人点赞："用户A 赞了你的评论"<br>- 2人点赞："用户A、用户B 赞了你的评论"<br>- 3人及以上："用户A、用户B、用户C 等X人赞了你的评论"（展示前3个用户名，X为总点赞人数）<br>每条通知必须附带被点赞评论的前 15 个字符作为预览。 |
| 通知分类 | 点赞通知属于"收到的赞"类别。                                 |
| 已读逻辑 | 用户进入消息中心页面时，所有通知默认未读。用户点击查看某条通知的详情后，该通知标记为已读。 |

#### F10：视频列表

| 需求项   | 规格                                                         |
| :------- | :----------------------------------------------------------- |
| 位置     | 应用首页（`/` 路径）。                                       |
| 数据源   | 仅展示状态为 `published` 的视频。                            |
| 排序     | 按以下优先级排序：<br>1. 播放量降序<br>2. 上传时间降序<br>3. 弹幕数降序 |
| 卡片内容 | 每个视频卡片展示：**封面、标题、播放量、弹幕数、视频时长、UP主、创建时间**。 |
| 分页     | 必须使用游标分页。接口返回下一页的游标值，前端根据此值判断是否还有下一页。 |

### 三、验收标准

| 编号      | 验收项                     | 达标标准                                                     |
| :-------- | :------------------------- | :----------------------------------------------------------- |
| **AC-1**  | 用户注册登录闭环           | 访问注册页面 → 输入用户名和密码 → 注册成功。使用该用户名和密码登录 → 成功获取 JWT Token。携带此 Token 访问一个需鉴权的接口 → 接口正常返回数据，非 401。 |
| **AC-2**  | 视频上传与转码闭环         | 登录后上传一个 100MB 的 MOV 视频文件 → 接口返回 201，视频列表中出现该视频，状态为"处理中"。等待转码完成后刷新 → 状态变为"已发布"，点击可正常播放 H.264 格式视频。 |
| **AC-3**  | 视频上传失败处理           | 上传一个故意损坏无法转码的视频文件 → 状态变为"转码失败"。上传者进入个人中心 → 能看到该视频，并显示具体失败原因。 |
| **AC-4**  | 视频列表多级排序与卡片信息 | 首页视频列表按以下优先级排序：第一级按播放量降序，第二级按上传时间降序，第三级按弹幕数降序。播放量相同的视频，新上传的排在前面；上传时间也相同的，弹幕数多的排在前面。每个视频卡片正确展示：封面、标题、播放量、弹幕数、视频时长、UP主、创建时间。 |
| **AC-5**  | 实时弹幕                   | 同一视频房间内，100 个用户同时在线发送弹幕。从任一用户点击发送，到另一用户屏幕上渲染出该条弹幕，总延迟 ≤ 200ms。 |
| **AC-6**  | 弹幕历史加载               | 打开任一视频时，系统通过 WebSocket 连接自动推送该视频最近的 200 条历史弹幕。前端按弹幕的 `video_time` 在视频对应时间点渲染显示。 |
| **AC-7**  | 弹幕敏感词审核             | 发送一条包含敏感词库中词汇的弹幕 → 后端返回明确的错误信息提示，弹幕不被广播，也不被存入数据库。 |
| **AC-8**  | 弹幕发送冷却               | 发送一条弹幕成功后，前端发送按钮立即变灰并显示 5 秒倒计时。倒计时期间通过任何方式尝试发送均被拒绝。倒计时归零后按钮恢复。 |
| **AC-9**  | 多级评论嵌套               | 在视频下方发表一级评论 → 成功。回复这条一级评论 → 生成二级评论。回复这条二级评论 → 生成三级评论。再次回复这条三级评论 → 新生成的评论仍为三级评论，其父评论指向被回复的那条三级评论。 |
| **AC-10** | 评论点赞与取消             | 对一条评论点赞 → 该评论点赞数 +1。同一个用户再次点击 → 点赞取消，点赞数 -1。 |
| **AC-11** | 点赞通知聚合               | 3 个不同用户对用户 B 的同一条评论点赞。用户 B 进入消息中心 → "收到的赞"分类下出现 **1 条**通知，内容为"用户A、用户B、用户C 等3人赞了你的评论"。该通知附带被点赞评论的前 15 个字符作为预览。 |
| **AC-12** | 消息中心结构               | 消息中心为独立页面，路由为 `/messages`。其左侧分类导航栏包含且仅包含五个选项：**回复我的**、**@我的**、**收到的赞**、**系统通知**、**我的消息**。每个分类旁显示该分类的未读消息数量角标。 |
| **AC-13** | 视频封面管理               | 上传视频时一并提交自定义封面图 → 转码完成后视频卡片和播放页展示该自定义封面。上传视频时不提交封面 → 系统自动生成第 1 秒帧作为默认封面。视频发布后通过封面修改接口上传新封面 → 封面被替换为新图。 |
| **AC-14** | 封面格式校验               | 上传视频时提交不符合要求的封面文件（格式非 JPEG/PNG/GIF/BMP/WEBP 或大小超过 10MB）→ 接口返回明确错误信息，整个上传请求被拒绝。 |
| **AC-15** | 评论删除权限               | UP主可以删除自己视频下的任意评论。普通用户可以删除自己的评论，但不能删除他人的评论。 |
| **AC-16** | 评论删除级联               | 删除一条有子评论的评论 → 所有子评论同步被删除，前端不再展示。 |

### 四、兼容要求

| 编号     | 类型     | 规格                                                         |
| :------- | :------- | :----------------------------------------------------------- |
| **BC-1** | 向后兼容 | 本版本为首个正式版本，无历史兼容负担。所有接口和数据模型定义后，后续版本未经评审不得对已有字段做破坏性修改。 |
| **BC-2** | 向前兼容 | 代码架构必须支持未来平滑拆分为 Kratos 微服务架构。v1.0 的单体实现不得引入阻碍后续拆分的耦合设计。视频数据表必须预留 `status` 字段以支持 `pending_review` 和 `rejected` 两种状态值，供后续版本审核功能使用。 |

### 五、非功能需求

| 编号 | 类型               | 刚性规格                                                     |
| :--- | :----------------- | :----------------------------------------------------------- |
| NF-1 | 并发量             | 弹幕系统必须支持同一视频房间内 100 人同时在线并发送弹幕，服务无崩溃、无死锁。 |
| NF-2 | 延迟               | 弹幕端到端延迟在本地开发环境必须 ≤ 200ms。                   |
| NF-3 | 存储划分           | MySQL：用户、视频元数据、视频状态、评论、点赞记录、通知记录。<br>Redis：弹幕实时通道中转、视频播放量热数据计数。 |
| NF-4 | 消息队列           | 视频转码任务必须使用消息队列异步投递和消费，不允许在上传请求中同步执行 FFmpeg。 |
| NF-5 | 前端架构           | 必须是 Vue 3 + Vite 构建的纯 SPA，不做 SSR。                 |
| NF-6 | 后端语言与核心约束 | 后端必须使用 Go 语言开发。所有代码必须遵循 Go 语言标准项目布局。代码架构必须支持未来平滑拆分为 Kratos 微服务，但 v1.0 版本不做强制拆分要求。 |
| NF-7 | 文件存储           | 所有文件存储在阿里云 OSS 的 **单一 Bucket** `mini-bili` 中，通过目录前缀区分文件类型：<br>- 转码后视频：`videos/{video_id}.mp4`<br>- 封面图（默认封面和自定义封面）：`covers/{video_id}.{ext}`<br>- **用户头像**：`avatars/{user_id}.{ext}`<br>原始视频文件及临时封面文件仅在上传和转码期间暂存于服务器本地临时目录，处理完成后立即清理，不上传至 OSS。 |
| NF-8 | API 风格           | 所有对外 HTTP 接口遵循 RESTful API 设计规范。                |
| NF-9 | 消息中心           | 消息中心必须是独立的一级页面（路由：`/messages`），包含左侧分类导航栏，导航项必须包含且仅包含五个：<br>- 回复我的<br>- @我的<br>- 收到的赞<br>- 系统通知<br>- 我的消息<br>点击不同分类可筛选对应类型的通知列表。导航栏需显示每个分类的未读数量角标。消息中心页面需要在导航栏的铃铛图标提供入口。 |
